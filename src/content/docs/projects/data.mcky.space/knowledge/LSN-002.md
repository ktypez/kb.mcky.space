# LSN-002: อัพรูปแล้ว save ไม่ได้ — base64 data URL เกิน limit Workers/D1

## อาการ
กดเพิ่ม/แก้ไขลูกค้า แล้วเลือกรูปถ่าย → กดบันทึก → **ทั้ง entry ไม่ถูกบันทึก** (เงียบ ไม่มี error แจ้งผู้ใช้)

## รากเหตุ

### 1. base64 ใน client payload → Workers 10MB request body limit
- `ImageUpload.tsx` อ่านไฟล์รูปเป็น **base64 data URL** แล้วเก็บใน `images: string[]`
- `storage.ts` ส่ง `JSON.stringify(client)` ไป POST/PUT /api/clients → base64 ทำให้ payload เกิน 10MB
- Request ล้ม → entry ไม่ถูก save

### 2. ขาด R2 bucket binding ใน wrangler.toml (ค้นพบรอบสอง)
- `wrangler.toml` มีเฉพาะ `[[d1_databases]]` ไม่มี `[[r2_buckets]]`
- `/api/photo-request` ใช้ `env.BUCKET` (R2 binding) และ `env.R2_PUBLIC_URL` ซึ่งเป็น undefined
- `uploadImage()` ใน r2.ts เรียก `bucket.put()` → TypeError → `Promise.allSettled` catch → คืน base64 เดิม
- ทำให้รูปไม่อัปโหลดไป R2, base64 ยังวนกลับไปเก็บใน D1

### 3. photo-request merge เก็บ base64 เก่าใน D1 ไว้ (ค้นพบรอบสอง)
- หลัง R2 upload สำเร็จ, photo-request merge existing images (จาก D1) กับ newUrls
- existing มี base64 strings จาก save ครั้งก่อน (ตอนที่ยังไม่มี R2 fix)
- merge = [existing base64, ...newUrls] → base64 ถูกเก็บไว้ใน D1 ต่อไป
- storage.ts เอาผล merge ไป PUT /api/clients → base64 ไปใน payload อีก → วน loop

### 4. react-router@7.18.1 import Node modules เพิ่มเรื่อยๆ
- หลังจาก fix `cookie` ไป (LSN-001), พบ `set-cookie-parser` import เข้า browser bundle
- ต้องสร้าง shim `src/shims/set-cookie-parser.js` ทุกครั้งที่เจอ module ใหม่

### 5. R2 object leak เมื่อลบลูกค้า (ค้นพบรอบสาม 2026-07-13)
- `onRequestDelete` ใน `functions/api/clients/[id].ts` ลบเฉพาะ record ใน D1
- **ไม่ลบรูปถ่ายที่อัปโหลดไว้ใน R2** → object ค้างเป็น orphan
- มี `deleteClientImages()` ใน `functions/lib/r2.ts` อยู่แล้วแต่ไม่ถูกเรียก
- ส่งผลให้จำนวน R2 objects > จำนวน clients (349 objects vs 324 clients)
- object ที่ค้างอยู่กิน storage โดยเปล่าประโยชน์ (แม้จะน้อย: ~295 KB/object)

## แนวทางแก้

### รอบแรก: แยก base64 images ออกจาก client payload
```
addClient(client):
  1. Filter isBase64Image() → base64Images, cleanImages
  2. POST /api/clients ด้วย { ...client, images: cleanImages } → ได้ id
  3. POST /api/photo-request → upload base64 ไป R2 → คืน R2 URLs
  4. Return client ที่ images = R2 URLs

updateClient(client):
  1. Filter isBase64Image() → base64Images, cleanImages
  2. ถ้ามี base64Images:
     a. GET /api/clients/[id] → prevImages (R2 URLs เดิม)
     b. POST /api/photo-request → upload base64 + ลบของเก่า → คืน merged URLs
  3. PUT /api/clients/[id] ด้วย images = finalImages
```

### รอบสอง: ปิดช่องโหว่ base64 leakage (2026-07-13)
1. **wrangler.toml**: เพิ่ม `[[r2_buckets]]` binding = "BUCKET", bucket_name = "ezzylist" และ `[vars]` R2_PUBLIC_URL
2. **photo-request.ts**: ตัด base64 ที่ upload แล้วออกจาก existing D1 merge
   ```
   const uploadedBase64 = new Set(images.filter(s => s.startsWith('data:image')))
   const kept = existing.filter(url => !deletedImages.includes(url) && !uploadedBase64.has(url))
   ```
3. **storage.ts**: filter base64 จาก finalImages เป็นชั้นป้องกัน
   ```
   if (result.ok) finalImages = result.images.filter(s => !isBase64Image(s))
   ```

### รอบสาม: R2 orphan cleanup เมื่อลบลูกค้า (2026-07-13)
- เพิ่ม import `deleteClientImages` จาก `../../lib/r2` ใน `functions/api/clients/[id].ts`
- ก่อนลบ record ใน D1, เรียก `deleteClientImages(env.BUCKET, env.R2_PUBLIC_URL, row.images)`
- เฉพาะ R2 URLs (ที่ขึ้นต้นด้วย http) จะถูกลบ, base64 strings จะถูกข้ามโดย `deleteClientImages` 

### compressImage — default maxSizeMB 10 → 3
- base64 overhead ~33%, 3MB JPEG → ~4MB base64 << 10MB
- 2 รูปรวมกัน ~8MB base64 ยังไม่เกิน limit

### shim set-cookie-parser
- `splitCookiesString` — split Set-Cookie header ตาม comma
- alias ใน vite.config.ts → `src/shims/set-cookie-parser.js`

## ขั้นตอนตรวจสอบ

### หลังรอบสอง deploy
1. `npx wrangler d1 execute ezzylist-db --remote --command "SELECT COUNT(*) FROM clients WHERE images LIKE '%data:image%'"`
   → ต้องได้ 0 (ไม่มี base64 ค้างใน D1)
2. ส่ง base64 ไป /api/photo-request → response ต้องมีแต่ R2 URLs
3. เช็ค D1 หลังจากนั้น → images field ต้องไม่มี base64

### หลังรอบสาม deploy
1. ลบลูกค้าที่มีรูปถ่าย → เช็ค R2 objects ว่าลดลง
2. `npx wrangler r2 bucket info ezzylist` → object_count ควรลดลงตามจำนวนที่ลบ

## ไฟล์ที่เกี่ยวข้อง
- `src/lib/storage.ts` — addClient/updateClient ส่ง base64 ไป R2 + filter base64 จาก finalImages
- `src/lib/compressImage.ts` — maxSizeMB default 10→3
- `src/shims/set-cookie-parser.js` — ESM shim สำหรับ set-cookie-parser
- `vite.config.ts` — alias set-cookie-parser
- `wrangler.toml` — R2 bucket binding + R2_PUBLIC_URL
- `functions/api/photo-request.ts` — strip uploaded base64 from D1 merge
- `functions/lib/r2.ts` — uploadClientImages, uploadImage (base64→R2), deleteClientImages
- `functions/api/clients/[id].ts` — onRequestDelete เรียก deleteClientImages ก่อนลบ D1 record
